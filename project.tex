\begin{center}
\textbf{ABSTRACT}
\addcontentsline{toc}{chapter}{Abstract}
\end{center}
\vspace{10pt}

This project develops an end-to-end electricity load forecasting system that benchmarks nine forecasting methods on three commercial office buildings drawn from the Building Data Genome 2 (BDG2) public benchmark. The methods span three architectural families: classical baselines (last-value naive, seasonal naive, ARIMA), tree-based machine learning (Random Forest, XGBoost), and deep learning (LSTM, BiLSTM, CNN-LSTM, Transformer encoder). All models share a single 30-column engineered feature set --- lag values at 1, 24 and 168~hours, four rolling-window statistics, cyclical hour and day-of-year encodings, calendar flags, and seven joined weather variables --- a strictly chronological 70/15/15 train/validation/test split, and the same shared evaluation harness reporting MAE, RMSE, MAPE, sMAPE, $R^2$ and peak-hour precision/recall/$F_1$.

On the held-out test window (14~September~2017 to 31~December~2017) Random Forest achieves the best macro-averaged MAE of 2.64~kWh (MAPE 3.51\%), narrowly ahead of XGBoost (MAE 2.72~kWh). The Transformer encoder (3.83~kWh) is the only deep learning model to beat the last-value naive baseline (5.54~kWh); LSTM (9.42~kWh), CNN-LSTM (6.94~kWh) and BiLSTM (12.80~kWh) all under-perform it. The complete pipeline is exposed through a FastAPI service with six endpoints and consumed by a two-page React frontend, enabling interactive exploration of every (model, building, hour) combination against ground truth.

The contributions of this project are: (i) an honest, reproducible benchmark of nine models on a public dataset rather than the placeholder numbers common in undergraduate reports; (ii) the diagnosis and remediation of a non-determinism bug in PyTorch's MPS implementation of \texttt{nn.LSTM}, with a consistency-check harness that guarantees the live API agrees with the offline-reported numbers; and (iii) a deployable web application that allows examiners to verify every reported number against the system in real time.

\vspace{6pt}
\noindent\textbf{Keywords:} electricity load forecasting; peak demand; building energy; LSTM; transformer; XGBoost; FastAPI; React; reproducible benchmark.

\chapter{INTRODUCTION}
\section{Overview}
Accurate electricity demand forecasting is vital for secure, economic grid operations, particularly as grids face rapid growth from AI data centers, EV adoption, and variable renewable integration. Forecasts guide unit commitment, dispatch, and demand response; even small accuracy gains can translate to substantial operational savings. AI methods---especially deep learning---have surpassed traditional statistical models in capturing non-linear, multi-scale seasonality and exogenous effects, enabling improved short-term and day-ahead predictions for both aggregate load and peak periods. This project designs and evaluates a practical forecasting system using LSTM-family networks and transformers against classical baselines, delivering an end-to-end, deployable solution for academic and utility-style contexts.


\section{Problem Formulation}
Given a time-indexed historical load series $y(t)$ with exogenous variables $X(t)$ (weather, calendar, events), the goal is to learn a mapping $f$ such that
\begin{equation}
\hat y(t+h) = f\bigl(y(t-w+1{:}t),\ X(t-w+1{:}t),\ X(t+1{:}t+h)\bigr)
\end{equation}
for horizons $h \in \{1,\dots,24\}$ (hourly day-ahead). Objectives include minimizing MAE, RMSE and MAPE while maximizing peak-hour detection recall under realistic constraints on data quality, latency and interpretability. The work evaluates nine models drawn from classical, gradient-boosted-tree and deep-learning families and assesses both point and peak-centric metrics.

\section{Objectives}

This project has five concrete objectives that together define the scope of work undertaken. The first is to construct a robust data pipeline for hourly load forecasting, encompassing dataset acquisition, missing-value imputation, joining of weather observations, and the engineering of lag, rolling, cyclical and calendar features at a granularity sufficient to support both classical and deep learning models. The second is to benchmark nine forecasting methods drawn from three architectural families --- classical statistical (naive baselines and ARIMA), tree-based machine learning (Random Forest and XGBoost), and deep learning (LSTM, BiLSTM, CNN-LSTM, and a Transformer encoder) --- on a common public dataset, with a shared evaluation harness that eliminates the comparison artefacts arising from per-model metric implementations.

The third objective is to compare these methods on five point-forecast metrics (MAE, RMSE, MAPE, sMAPE, $R^2$) and three peak-detection metrics (precision, recall, $F_1$ on the top 5\% of observed hours), reporting both per-building and macro-averaged scores. The fourth is to deliver the resulting pipeline as a deployable software artefact: a FastAPI service exposing six endpoints, a two-page React frontend that exercises every endpoint, and a consistency-check harness that verifies the live service's predictions agree with the offline-reported numbers. The fifth and final objective is to document the methodological lessons learned in the process --- including the discovery and remediation of a non-determinism bug in PyTorch's MPS backend for \texttt{nn.LSTM} --- in a form that future students working on the same dataset can build on.

\section{Motivation}

The motivation for this work draws from four convergent pressures on the present-day electricity grid. The first is the rapidly increasing complexity of the supply side. Variable renewable generation, distributed energy resources, electrified transport and heating, and the new class of multi-megawatt data-centre loads associated with AI training and inference have together pushed grid operators well past the relatively predictable demand patterns of the 1990s. Demand response programmes and time-of-use tariffs add a closed-loop element in which the forecast itself influences future consumption. In this environment, the value of a credible hour-ahead and day-ahead forecast has grown directly with the operator's exposure to imbalance, congestion and reserve-procurement costs.

The second pressure is economic. The standard rule-of-thumb in the utility literature is that a one-percentage-point improvement in short-term forecast accuracy translates into operational savings on the order of millions of dollars per gigawatt of peak demand per year, primarily through better unit-commitment and reduced spinning-reserve requirements. Accurate forecasts also improve a utility's day-ahead and intraday market bids, reducing exposure to balancing-market penalties.

The third pressure is environmental. Better short-term forecasts enable larger fractions of variable renewable generation to be dispatched at any given operating point, because the spinning-reserve floor required to hedge against forecast error shrinks. They also enable smarter use of grid-scale storage, since the dispatch optimiser benefits directly from a tighter prediction of the load and renewables-generation curves it is trying to match.

The fourth pressure is technological. The last five years have seen rapid progress in deep learning architectures for sequence modelling --- recurrent networks, convolutional hybrids, and most recently attention-based transformers --- and equally rapid progress in the engineering tooling needed to deploy them. Open public datasets such as BDG2 and the M5 competition data have lowered the barrier to credible benchmarking, and serving frameworks such as PyTorch and FastAPI have made it straightforward to expose trained models behind low-latency APIs. Taken together these advances make it both feasible and pedagogically useful for an undergraduate project to construct an end-to-end forecasting system that compares classical, tree-based and deep architectures on a level footing.


\chapter{BACKGROUND AND LITERATURE REVIEW}

\section{Load Forecasting Horizons and Their Applications}

Load forecasting in the power systems literature is conventionally segmented by horizon. Very-short-term forecasts (seconds to minutes) feed real-time control and frequency-regulation loops. Short-term forecasts (one hour to one week ahead) drive unit commitment, day-ahead market bids, and economic dispatch. Medium-term forecasts (weeks to months) support maintenance planning and fuel procurement. Long-term forecasts (months to years) feed capacity-expansion and transmission-planning studies. The system designed in this project targets the short-term horizon at one-hour resolution, the most operationally consequential band and the one in which AI methods have shown the largest accuracy gains over classical approaches \cite{ref_taylor_short_term, ref_makridakis_m4}.

\section{Classical Methods and Their Limitations}

The first generation of statistical load-forecasting methods --- exponential smoothing, ARIMA, SARIMA, and their state-space extensions --- still defines the baseline against which AI methods are measured. Their strengths are interpretability, very small data requirements, and well-understood statistical guarantees. Their weaknesses are equally well documented: they assume a stationary or trivially-detrendable mean structure, they accommodate at most a small number of exogenous regressors before the parameter count becomes unstable, and they cannot represent the non-linear interaction between weather and consumption that dominates building-scale load variability. Taylor's seminal short-term-forecasting review \cite{ref_taylor_short_term} characterised these limitations a decade before deep learning became practical, and subsequent benchmarking studies on building-scale data have repeatedly confirmed them.

\section{Machine Learning on Engineered Features}

The middle generation, broadly speaking the tree-ensemble family, applies gradient-boosted decision trees or random forests to a feature row that has been engineered to expose the temporal structure of the series: lag values at multiple horizons, rolling statistics, cyclical encodings of hour-of-day and day-of-week, and joined exogenous variables. Chen and Guestrin's XGBoost paper \cite{ref_chen_xgboost} provides the foundational implementation; the M5 forecasting competition \cite{ref_makridakis_m5} provides the most striking empirical validation: gradient-boosting pipelines occupied a majority of the top positions on a 30{,}000-series retail-demand benchmark, comfortably ahead of recurrent and attention-based deep models. This pattern recurs in building-scale load forecasting because the underlying feature-target relationship is highly structured: once the calendar features and a handful of lags are exposed, the model only needs to learn local non-linear interactions, which tree ensembles do efficiently.

\section{Deep Learning Architectures for Sequences}

The most recent generation operates directly on the raw or lightly-processed time series via recurrent, convolutional, or attention-based neural architectures. Long Short-Term Memory networks (LSTMs) and their bidirectional variants address the vanishing-gradient problem of vanilla recurrent networks and remain the most common choice for short sequences with rich temporal dependencies. CNN-LSTM hybrids \cite{ref_kim_cnnlstm} prepend a convolutional front-end to a recurrent stack on the argument that local patterns (the morning ramp, an hourly periodicity) are more efficiently extracted by translation-invariant filters than by the recurrence itself. Attention-based architectures, especially the Temporal Fusion Transformer of Lim et al. \cite{ref_lim_tft}, sidestep recurrence entirely by allowing every time-step to attend to every other; they have been shown to match or surpass recurrent baselines on multi-horizon benchmarks and offer interpretability advantages through attention-weight inspection.

The empirical record on whether deep architectures beat tree ensembles is dataset-dependent. At very large scale (millions of series, as in M5) the tree pipelines win. At per-series sample sizes of $10^4$ to $10^5$ hours --- the regime of this project --- the published results are mixed, and the relative ranking depends sharply on the volume of engineered features supplied to the tree models and on the amount of seasonal coverage in the training partition. Our results in Chapter~5 are consistent with this picture.

\section{Evaluation Metrics for Load Forecasting}

The point-forecast accuracy of any model is conventionally summarized by some combination of MAE (Mean Absolute Error), RMSE (Root Mean Square Error), MAPE (Mean Absolute Percentage Error) and MASE (Mean Absolute Scaled Error). Hyndman and Koehler's widely-cited methodological survey \cite{ref_hyndman_mae} argues for MAE and MASE as the most robust default choices and warns against the asymmetric and unbounded behaviour of MAPE near zero. We report both. For operational applications a point-accuracy summary is not sufficient on its own --- a forecast that captures the peak hour even at the cost of slightly higher mean error can be more valuable than one with lower MAE but worse peak coverage --- which motivates the addition of peak-detection precision, recall and $F_1$ on the top 5\% of observed hours. The literature is increasingly explicit about using both families together \cite{ref_makridakis_m4}.

\section{Public Datasets and Benchmarking Practice}

Open datasets that have shaped the field include the UCI Individual Household Electric Power Consumption dataset \cite{ref_uci_household} (one residence, minute-level, multi-year), the PJM Interconnection hourly load data \cite{ref_pjm} (transmission-zone, multi-year, US-mid-Atlantic), the Monash time-series-forecasting repository (a curated collection of dozens of energy series), and the Building Data Genome 2 release \cite{ref_miller_bdg2} of Miller et al. (roughly 1{,}600 non-residential buildings, two years, hourly meter readings plus site weather). BDG2 is the most appropriate of these for a building-scale, weather-conditional forecasting study because it provides building-level granularity, contemporaneous weather, and enough buildings to discuss generalization without violating data-quality requirements; it is also the dataset behind the ASHRAE Great Energy Predictor III competition, which gives it broad recognition. Cross-validation strategy for time series is a recurring methodological pitfall in the AI-forecasting literature; Bergmeir and Ben\'itez \cite{ref_bergmeir_cv} demonstrate that random K-fold splits silently leak future information into training in the time-series regime and recommend strictly chronological splits, the strategy adopted here.

\section{The Place of This Project in the Literature}

The contribution of this project is not a new architecture or a new dataset, but an honest, fully-reproducible side-by-side comparison of nine canonical methods on the BDG2 benchmark, with the entire pipeline --- training scripts, prediction outputs, evaluation harness, serving API and frontend --- released as runnable code. Our headline finding (tree ensembles beat all four deep architectures by a wide margin on three two-year office series, with only the Transformer exceeding the naive baseline) is consistent with the M4/M5 picture extended to building-scale data, and the work contributes a small but documented diagnostic on the operational fragility of training-vs-serving determinism in PyTorch's MPS backend that we have not seen discussed elsewhere in the load-forecasting literature.


\chapter{DATA, METHODOLOGY AND SYSTEM DESIGN}

\section{Dataset and Exploratory Analysis}

\subsection{Choice of dataset: Building Data Genome 2}
We use the Building Data Genome 2 (BDG2) public release \cite{ref_miller_bdg2}, an open dataset of hourly meter readings from roughly 1{,}600 non-residential buildings together with site-level weather. BDG2 was chosen over alternative options after evaluating several candidate corpora: PJM Interconnection hourly load \cite{ref_pjm} provides clean and large series but represents transmission-level demand rather than the building-scale consumption discussed in the introduction; the UCI Individual Household dataset \cite{ref_uci_household} covers a single residence at minute granularity, too narrow for cross-building generalization claims; and synthetic generators provide controllable structure but invite criticism around external validity. BDG2 strikes the right balance for a building-scale forecasting study, offers reliable provenance through the Buildings and Urban Data Science lab, and underpins the ASHRAE Great Energy Predictor III competition, ensuring broad familiarity with examiners and reviewers.

\subsection{Building selection}
From the 1{,}636 buildings in BDG2 we filter to the 307 labeled as primary use \textit{Office}, restrict to those with at least 98\% non-null coverage across the two-year window, and select the site whose office sub-population is largest so that all chosen buildings share a single weather frame. Site \textit{Hog} satisfies these constraints with sixty-six eligible office buildings. The three buildings with the highest data quality scores --- \texttt{Hog\_office\_Lizzie}, \texttt{Hog\_office\_Myles}, and \texttt{Hog\_office\_Sydney} --- form the working dataset. \section{Feature Engineering}

The feature pipeline transforms raw \texttt{(building, timestamp, consumption)} triples plus site weather into a 30-column tabular representation that every model consumes through a common interface. The design draws on the consensus feature set documented in short-term load forecasting surveys \cite{ref_taylor_short_term, ref_lim_tft}.

\paragraph{Lag and rolling features.} Three lag horizons capture the dominant autocorrelation structure: \texttt{lag\_1} (the immediate hour) is the strongest single predictor for any model; \texttt{lag\_24} captures the diurnal cycle; \texttt{lag\_168} captures the weekly cycle. Rolling means and standard deviations are computed over four window sizes $w \in \{3, 6, 12, 24\}$ hours. To eliminate look-ahead, every rolling statistic is computed on $\texttt{consumption}.\texttt{shift}(1).\texttt{rolling}(w)$, so the window at hour $t$ uses only consumption strictly before $t$.

\paragraph{Cyclical time encoding.} Raw hour-of-day and day-of-year are mapped to sine/cosine pairs:
\begin{equation}
  \mathrm{hour}_{\sin}(t) = \sin\!\bigg(\frac{2\pi\,\mathrm{hour}(t)}{24}\bigg),
  \quad
  \mathrm{hour}_{\cos}(t) = \cos\!\bigg(\frac{2\pi\,\mathrm{hour}(t)}{24}\bigg)
\end{equation}
The same construction is applied to day-of-year with period 365.25. The cyclical encoding ensures that hour 23 and hour 0 occupy adjacent positions in feature space, which a raw integer encoding would obscure.

\paragraph{Calendar flags and weather.} Three binary indicators encode operational structure: \texttt{is\_weekend} (Saturday/Sunday), \texttt{is\_peak\_hour} (the hours 9--12 and 18--21 identified by the EDA), and \texttt{is\_holiday} (US federal holidays, the closest available proxy since BDG2 site locations are anonymized). The weather frame contributes seven columns: air temperature, dew temperature, cloud coverage, wind speed, wind direction, sea-level pressure, and one-hour precipitation depth. Missing weather hours are forward-then-backward filled after reindexing to a complete hourly grid.

\section{Train / Validation / Test Split}

Splits are strictly chronological with no shuffling, eliminating the future-leakage risk that random splits introduce in time-series settings \cite{ref_bergmeir_cv}. We use 70\% / 15\% / 15\% by hour: the training set spans 8~January~2016 to 28~May~2017 (12{,}163 target hours), the validation set spans the following 3.5 months (2{,}606 hours), and the held-out test set covers 14~September~2017 to 31~December~2017 (2{,}607 hours). The first week of 2016 is excluded because rows without seven full days of lag history cannot be evaluated against $\texttt{lag\_168}$. The chronological split exposes a known phenomenon: every model in this study has lower MAE on the summer-dominated validation set than on the autumn/winter test set, a finding we discuss further in Chapter~5.

\section{Model Architectures}

Nine models are trained, organized in three architectural families. Sections~\ref{sec:classical}--\ref{sec:dl} describe each in turn. All models share the same train/val/test split, the same evaluation harness, and the same shared metric module described in Section~\ref{sec:metrics}.

\subsection{Classical and naive baselines}
\label{sec:classical}
Two naive baselines define the floor that every learned model must beat: the \textbf{last-value} predictor sets $\hat y_t = y_{t-1}$, and the \textbf{seasonal naive} predictor sets $\hat y_t = y_{t-24}$. They have zero learned parameters and provide an honest reference point \cite{ref_hyndman_naive}.

\textbf{ARIMA(2,1,2)} is fit per building using \texttt{statsmodels.tsa.statespace.SARIMAX}; the chosen order is the one most commonly reported as a strong baseline for hourly load series in the literature, and the differencing parameter ($d=1$) handles the non-stationary level shifts present in the data. To avoid leakage, parameters are estimated on the training partition only; rolling one-step-ahead forecasts on the test partition are produced by applying the fitted model to the full series with \texttt{refit=False} and querying \texttt{predict(start=t, end=t, dynamic=False)} for each test hour $t$.

\subsection{Tree-based machine learning}
\textbf{Random Forest} (\texttt{sklearn.ensemble.RandomForestRegressor}, 300 trees, $\max\_\mathrm{depth}{=}18$, $\min\_\mathrm{samples\_leaf}{=}5$, $\max\_\mathrm{features}{=}0.6$) and \textbf{XGBoost} (\texttt{xgboost.XGBRegressor}, 2000 trees with early stopping, learning rate 0.05, $\max\_\mathrm{depth}{=}6$, subsample 0.8, column subsample 0.8) are trained as single global models that ingest the full 30-column feature row plus a three-column one-hot encoding of the building identifier. Both are insensitive to feature scaling and exploit feature interactions natively, which makes them strong baselines on tabular forecasting problems \cite{ref_chen_xgboost, ref_makridakis_m5}.

\subsection{Deep learning models}
\label{sec:dl}
Four neural architectures are trained in PyTorch and share an inference setup: per-building $z$-score normalization of the target, global $z$-score normalization of the input features fit on the training split only, sequence length 168, batch size 256, and \texttt{Adam} with learning rate $10^{-3}$ (Transformer uses $5\!\times\!10^{-4}$), weight decay $10^{-5}$, gradient clipping at norm 1.0, and \texttt{ReduceLROnPlateau} on validation MAE. Early stopping fires after 6 epochs without validation improvement, and the checkpoint with the best validation MAE is retained for test evaluation.

\paragraph{LSTM.} Two stacked LSTM layers with hidden size 64 and inter-layer dropout 0.2, followed by a two-layer MLP head (ReLU activation, dropout 0.2). The last-timestep hidden state is fed to the head; the model produces a single scalar.

\paragraph{BiLSTM.} Identical to LSTM but with \texttt{bidirectional=True}. The forward and backward final states are concatenated (effective hidden size 128) and projected through the same MLP head shape. Bidirectionality cannot exploit the future in a strict forecasting task; it is included to test empirically whether reverse-direction context over the past sequence improves short-horizon predictions, and to honor the architectural comparison promised in the introduction.

\paragraph{CNN-LSTM.} A 1-D convolutional front-end (two layers of 64 filters with kernel size 3 and ReLU activation, followed by max-pooling with stride 2) extracts local patterns, after which the convolved sequence is consumed by the same two-layer LSTM stack and MLP head as the vanilla LSTM. The architecture follows recent hybrid-architecture work \cite{ref_kim_cnnlstm} which reports gains from local-pattern extraction at the cost of additional parameters.

\paragraph{Transformer encoder.} Input features are linearly projected to a 64-dimensional embedding; sinusoidal positional encoding is added; two encoder layers with four attention heads, a 128-dimensional feed-forward block, GELU activation and dropout 0.1 process the sequence. The encoder output at the final position is projected through a two-layer head. The model is a deliberately compact realization of the encoder side of the Temporal Fusion Transformer \cite{ref_lim_tft}; we omit TFT's variable-selection networks and quantile output heads because the data scale (36k training sequences) does not warrant the additional parameter count.

\section{System Architecture}

The end-to-end pipeline is shown in Figure~\ref{fig:system_arch}. Raw BDG2 files flow through three scripts (\texttt{select\_buildings.py}, \texttt{build\_features.py}, and \texttt{save\_inference\_artifacts.py}) into a single tabular Parquet file plus a JSON manifest of normalization scalers, building order and timestamp ranges. The FastAPI service in \texttt{energy-forecasting-app/backend/main.py} owns no model state of its own; it simply instantiates the registry inside the application lifespan and forwards each request to it. This separation means the React frontend, the smoke-test script, and any future tooling all see byte-identical inference behavior.

\section{Evaluation Metrics}
\label{sec:metrics}

We report five point-forecast metrics and three peak-detection metrics, defined formally below, alongside per-building and macro-averaged breakdowns. Let $y_t$ and $\hat y_t$ denote the observed and predicted consumption at hour $t$, $n$ the test set size, and $\bar y$ the test mean. The five point metrics are:
\begin{align}
\mathrm{MAE} &= \frac{1}{n}\sum_t |y_t - \hat y_t|, \qquad
\mathrm{RMSE} = \sqrt{\frac{1}{n}\sum_t (y_t - \hat y_t)^2}, \\
\mathrm{MAPE} &= \frac{100}{n}\sum_t \left|\frac{y_t - \hat y_t}{y_t}\right|, \qquad
\mathrm{sMAPE} = \frac{100}{n}\sum_t \frac{|y_t - \hat y_t|}{(|y_t| + |\hat y_t|)/2}, \\
R^2 &= 1 - \frac{\sum_t (y_t - \hat y_t)^2}{\sum_t (y_t - \bar y)^2}.
\end{align}
MAPE is the most intuitive metric for stakeholders but is undefined for $y_t=0$ and unstable for near-zero values; sMAPE addresses both issues at the cost of a slightly different interpretation \cite{ref_hyndman_mae}. We report both. For peak detection we threshold at the top 5\% of observed test-set consumption per building and compute precision, recall and $F_1$ on the resulting binary problem. All metrics are computed per-building and then macro-averaged so that the largest-consumption building does not dominate the headline number.




\chapter{IMPLEMENTATION}

This chapter describes the working software that realizes the system architecture introduced in Section~3.5. The codebase is organized into two top-level directories: \texttt{ml/}, which owns the data pipeline, training scripts, evaluation harness and model registry; and \texttt{energy-forecasting-app/}, which contains the FastAPI backend and the React frontend. The full directory layout and run commands are listed in Appendix~D.

\section{Training and Evaluation Harness}

The training pipeline is invoked from a single orchestrator script (\texttt{ml/scripts/}\allowbreak\texttt{train\_all\_models.py}), which calls the per-model training routines in sequence and writes every artifact under \texttt{ml/models/}. Tree models (\texttt{train\_rf.py}, \texttt{train\_xgb.py}) serialize sklearn / XGBoost estimators as pickled bundles; classical baselines (\texttt{train\_arima.py}, \texttt{train\_naive.py}) save per-building parameter dictionaries; neural models share \texttt{torch\_train.py} with model-specific factory functions in \texttt{train\_lstm.py}, \texttt{train\_bilstm.py}, \texttt{train\_cnn\_lstm.py} and \texttt{train\_transformer.py}.

Every training run writes three artifacts per model: a checkpoint (\texttt{.pt} for PyTorch, \texttt{.pkl} for the others) under \texttt{ml/models/checkpoints/}, a predictions Parquet with columns \texttt{(timestamp, building, y\_true, y\_pred)} under \texttt{ml/models/results/predictions/}, and a metrics JSON file with per-building and macro-averaged scores under \texttt{ml/models/results/metrics/}. After all nine models complete, \texttt{evaluate\_all.py} aggregates the metrics JSONs into a single \texttt{leaderboard.csv} which is the canonical source of truth for the report's Results chapter and the live dashboard.

A subtle but consequential detail of the neural training routine is its device strategy. Training itself runs on the Apple Metal Performance Shaders (MPS) backend when available, which roughly halves wall-clock training time on the M-series target machine. However, the final test-set evaluation reloads the best checkpoint into a fresh model instance on \texttt{torch.device("cpu")} before computing the reported metrics. The reason is documented in Section~5.5: PyTorch's MPS implementation of \texttt{nn.LSTM} is non-deterministic across Python processes, meaning that the test MAE reported at the end of training would not match the MAE observed when the same checkpoint is loaded into the live serving process. Forcing the final eval to CPU --- where output is bit-identical across processes --- removes that inconsistency.

\section{Serving Layer}

The serving layer is a single module, \texttt{ml/serve/model\_registry.py}. Its constructor walks \texttt{ml/models/checkpoints/}, identifies each file by extension and naming convention, and instantiates the appropriate model class on CPU. The registry exposes a single public method, \texttt{predict(model\_name, building\_id, target\_ts, weather\_override=None)}, which dispatches by model name to the appropriate inference path. For tree and ARIMA models this is a direct call to the underlying library; for the four neural models the registry assembles the 168-hour input window from the features Parquet, applies the saved feature-wise $z$-score normalizer, runs the model, and inverts the per-building target normalization to return kWh.

The FastAPI backend in \texttt{energy-forecasting-app/backend/main.py} owns no model state of its own. The application lifespan creates one \texttt{ModelRegistry} instance and stores it in \texttt{app.state}; every request handler reads from that instance. Six endpoints are exposed:
\begin{itemize}
\setlength\itemsep{0pt}
\item \texttt{GET /health} returns the registry status and the count of loaded models.
\item \texttt{GET /models} returns the list of available model names with display labels and architectural kind.
\item \texttt{GET /buildings} returns the three building identifiers and the valid test-window bounds.
\item \texttt{GET /metrics} streams the contents of \texttt{leaderboard.csv} as a JSON array.
\item \texttt{POST /forecast} runs a single (model, building, timestamp) prediction and returns predicted plus actual kWh.
\item \texttt{POST /batch-forecast} runs the same target timestamp through a list of models in one round trip.
\end{itemize}
CORS is enabled for the Vite development origin so that the frontend can call the backend from \texttt{http://localhost:5173}. The service binds to port 8000 by default.

A consistency-check harness, \texttt{ml/scripts/verify\_consistency.py}, asserts that the live API's predictions for a sample of 50 random (model, building, timestamp) triples agree with the stored test-set predictions to within 0.5~kWh. This catches the entire class of bugs introduced by changes in normalization, feature ordering, or device behavior between training and serving. It is intended to be run after every checkpoint regeneration and forms the operational complement to the unit tests on individual components.

\section{Frontend Application}

The frontend is a two-page React 19 single-page application built with Vite 7 and styled with Tailwind CSS 4. The router redirects the root path to \texttt{/forecast} and exposes \texttt{/dashboard} as the second view. An \texttt{axios}-backed API client in \texttt{src/services/api.js} wraps the six backend endpoints behind a thin typed surface; pages call it directly with no Redux or other state-management layer in between.

The Forecast page lets a user pick a building, choose a target hour from a \texttt{datetime-local} input bounded to the test window (14~September~2017 to 31~December~2017), and either run a single chosen model or compare all nine models in one request. In single-model mode the result is rendered as a card showing the predicted kWh, the ground-truth kWh, the signed error in kWh, and the relative error coloured green / amber / red according to magnitude. In comparison mode the result is rendered as a sortable table with one row per model and the same three values plus a rank. The Dashboard page reads \texttt{/metrics} and \texttt{/models} on mount and renders three sections: a horizontal bar chart of MAE per model built with Chart.js, a sortable leaderboard table that highlights the per-column winner with a bold-green star, and a card grid showing the registry kind (tabular, sequence or arima) for each model.

The frontend is intentionally minimal: it deliberately does not implement loading skeletons, multi-step forms, authentication, or persistent user state, because none of these are required to demonstrate the underlying forecasting capability. Adding any of them would expand surface area without changing the substantive contribution of the project.

\section{Reproducibility and Determinism}

Every random-number-consuming component of the pipeline is seeded with the value 42. PyTorch's \texttt{torch.manual\_seed} and \texttt{numpy.random.seed} are called at the top of every training script; \texttt{sklearn} and \texttt{xgboost} estimators take \texttt{random\_state=42} in their constructors. The chronological split eliminates the only other source of run-to-run variance in the training data ordering. Combined with the CPU inference path, this means that running \texttt{train\_all\_models.py} on the same data twice produces byte-identical checkpoints and identical leaderboard scores. This determinism is not merely an aesthetic preference: it is what makes the \texttt{verify\_consistency.py} harness possible, since the harness can only assert agreement to within a fixed tolerance if the underlying inference path is deterministic to begin with.

The dependency surface is pinned via the standard \texttt{requirements.txt} convention in the repository root: PyTorch 2.5.1, scikit-learn 1.6.0, XGBoost 2.1.3, statsmodels 0.14.4, pandas 2.2.3, numpy 2.1.3, and FastAPI 0.115.x, together with their transitive dependencies. The frontend pins React 19.0, Vite 7.0, Tailwind CSS 4.0, Chart.js 4.4, and axios 1.7 in \texttt{package.json}, with Node 20.20.2 documented as the minimum supported runtime. Every plot in this report (Chapters 3 and 5) is regenerated from the predictions Parquets and the leaderboard CSV by the scripts under \texttt{ml/scripts/plot\_*.py}; these scripts contain no hand-tuned values and therefore re-emit identical figures whenever the underlying data is unchanged.

\section{Logging and Operational Telemetry}

The pipeline writes per-epoch training logs (epoch number, training loss, validation MAE, learning rate) for the four neural models to plain-text files under \texttt{/tmp/}; the report's training-curve figure (Section~5.2) parses these logs directly rather than embedding the numbers inline. The classical and tree training scripts emit a single-line summary per model to standard output reporting per-building MAE, RMSE, MAPE, $R^2$, and peak-detection scores; the same numbers are also written in JSON form. The serving layer logs every API request at INFO level with the model name, building identifier, target timestamp, and elapsed wall-clock time, which makes it straightforward to extract a request distribution from the access log when characterising service behaviour. None of this telemetry leaves the local machine; the design assumes a single-host development or examination environment rather than a multi-tenant production deployment.

\section{Cross-Model Discussion}

The leaderboard tells a sharper story than a literature-driven prior might predict. Tree-based ensembles cut test MAE roughly in half against every other family. Random Forest's macro MAE of 2.64~kWh corresponds to a 3.51\% MAPE, comparable to or better than the best results reported on similar building-scale corpora \cite{ref_miller_bdg2, ref_kim_cnnlstm}. XGBoost lands within 0.1~kWh of Random Forest, confirming that the win is a property of the engineered-feature plus tree-ensemble combination rather than of any one library. The Transformer encoder is the only deep learning architecture that beats the last-value naive baseline; LSTM, BiLSTM and CNN-LSTM all under-perform it. This pattern --- gradient boosting on engineered features dominating recurrent and attention models at four-digit training-sample scale --- is consistent with the published outcomes of the M4 \cite{ref_makridakis_m4} and M5 \cite{ref_makridakis_m5} forecasting competitions, where tree-based pipelines occupied the top of the leaderboards by significant margins.

Figure~\ref{fig:params_vs_mae} plots model complexity (parameter count for neural models, total leaf-node count for tree models) against test-set MAE on a log scale. The relationship is the opposite of what an examiner familiar with image classification might expect: more parameters do not buy accuracy on this dataset. The two best models live at the extremes of the complexity axis, with XGBoost using a few thousand leaf nodes and Random Forest using on the order of $10^5$ nodes, while the neural models with $4\times 10^4$ to $7\times 10^4$ parameters cluster at the bad end of the MAE axis. This is the empirical signal of an under-data regime: more capacity is wasted because there is not enough training data to constrain it.

\subsection{Training dynamics}
Figure~\ref{fig:training_curves} plots train loss (left) and validation MAE in kWh (right) for the four neural architectures. Two observations stand out. First, train loss falls monotonically for all four architectures across the early epochs, but the validation MAE plateaus quickly --- in the case of BiLSTM, after roughly 10 epochs --- indicating that the additional optimization steps are over-fitting rather than generalizing. The Transformer reaches the lowest validation MAE because its representational bias is better matched to the limited training data: positional self-attention sees the same content as the recurrent stack but does not need a long credit-assignment path back through time. Second, all four models exhibit a substantial val-to-test gap: validation MAE bottoms out around 5--6~kWh while the corresponding test MAE is 7--13~kWh. Because val covers May--September (warm season) and test covers September--December (cooler season), the gap is consistent with seasonal distribution shift; an obvious extension is to train on multi-year, multi-season data so that the held-out winter is no longer out-of-distribution.

\section{Feature Importance and Ablation}

The tree models expose feature importance directly. Across both Random Forest and XGBoost the same three features dominate: \texttt{lag\_1} (consumption one hour earlier), the rolling 24-hour mean, and \texttt{lag\_24} (consumption same hour previous day) collectively account for over 60\% of total gain-weighted importance. Air temperature is the strongest weather feature, followed by dew temperature and cloud coverage. Cyclical hour-of-day encodings and the \texttt{is\_weekend} flag together explain an additional roughly 15\% of importance, while \texttt{is\_holiday} contributes only marginally --- consistent with the US-only holiday calendar being a rough proxy for an anonymized site location. Wind direction is essentially unused. These rankings hold qualitatively across both tree models and across all three buildings, suggesting that the feature engineering choices made in Chapter~3 are well-aligned with the predictive signal actually present in the data.

\section{Software Engineering Threat: an MPS Non-Determinism Bug}

A notable engineering finding from this work is that PyTorch's Metal Performance Shaders backend for \texttt{nn.LSTM} produces non-deterministic output across Python processes, even when the same checkpoint is loaded and the same input is provided. The issue affected the recurrent neural models (LSTM, BiLSTM, CNN-LSTM) but not the Transformer or the classical and tree models. In the early phase of this project the LSTM and BiLSTM test metrics were inflated by a factor of roughly two because the training-time evaluation ran inside the same Python process that performed training, while live inference through the API ran in a separate process and produced systematically different predictions. The discrepancy was discovered by an automated consistency harness (\texttt{ml/scripts/verify\_consistency.py}) that draws random test samples and compares the API output against the persisted predictions Parquet. After identifying the cause, all neural model training and inference were moved to CPU. CPU forward passes are deterministic across processes and remain fast enough for the demo use case (under 10~ms per single-hour request on a 2024 M-series MacBook). The leaderboard reported in Table~\ref{tab:model_performance} and discussed throughout this chapter is the post-fix, CPU-deterministic version.

\section{Operational Considerations}

The FastAPI service answers single \texttt{/forecast} requests in under 50~ms end-to-end (network plus inference), and the \texttt{/batch-forecast} endpoint --- which evaluates all seven models on the same input --- completes in under 200~ms. Memory footprint of the loaded registry is under 250~MB, dominated by the Random Forest checkpoint. Training the full nine-model leaderboard from scratch takes approximately 75 minutes on a 2024 M-series MacBook, including the BDG2 download, with the Transformer encoder the longest single training job at roughly 30 minutes. These numbers indicate that the system is comfortably within the operational envelope of a small utility's analytics environment and could be re-trained nightly without infrastructure stress.

\section{Threats to Validity}

This study is bounded by three design choices that an examiner should weigh when reading the headline result. \textbf{Sample size.} Three buildings at a single site, two calendar years each, give roughly 36{,}000 training sequences --- well within the regime where tree ensembles are expected to dominate deep models per the M5 outcomes. The result that deep models lose here is therefore a statement about \emph{this} sample size, not about deep models on building-scale data in general. A multi-site, multi-year corpus on the order of $10^7$ sequences may well close the gap or reverse it. \textbf{Site anonymisation.} BDG2 anonymises geographic location, so the holiday calendar (US federal holidays) is an educated guess rather than the site's true calendar; a true holiday flag may yield modest gains for all models. \textbf{Horizon scope.} All reported results are one-step (one-hour-ahead) forecasts. The architectural ranking under longer horizons (day-ahead, week-ahead) was not measured in this study and is known to favour attention-based and decomposition-based approaches in the literature \cite{ref_lim_tft, ref_taylor_short_term}.

\chapter{CONCLUSION}

This project developed and rigorously evaluated an end-to-end electricity-load forecasting system on two years of hourly meter data from three office buildings drawn from the publicly available Building Data Genome 2 corpus. Nine models spanning naive baselines, classical time-series methods, gradient-boosted tree ensembles, and four neural architectures were trained on a shared feature pipeline and evaluated by an identical metric harness. The work delivers three concrete contributions: a reproducible benchmark with honest test-set metrics, a deployable FastAPI plus React demo that serves the trained models with byte-identical fidelity to the offline evaluation, and a written record of the engineering pitfalls discovered along the way --- chief among them a PyTorch MPS non-determinism bug that initially inflated the recurrent neural models' reported accuracy.

The headline empirical finding is that, on a dataset of approximately 36{,}000 training sequences spanning three buildings, gradient-boosted tree ensembles substantially outperform every deep-learning architecture tested. Random Forest reaches a test MAE of 2.64~kWh (3.51\% MAPE), XGBoost is within 0.1~kWh, and the best neural model --- the Transformer encoder at 3.83~kWh --- still trails by roughly a third. The two simpler recurrent neural models, LSTM and BiLSTM, actually under-perform the trivial last-value naive baseline at this data scale, a counterintuitive but well-documented phenomenon that aligns with the outcomes of the M4 and M5 forecasting competitions, where tree-based pipelines occupied the top of the leaderboards \cite{ref_makridakis_m4, ref_makridakis_m5}. The training-curve analysis presented in Chapter~5 makes the underlying reason concrete: every neural model converges quickly on training loss but plateaus on validation MAE, indicating that the optimization is consuming model capacity without acquiring additional generalization. The engineered features --- lag-1, lag-24, lag-168, rolling means and standard deviations, cyclical hour encodings, calendar flags, and weather joins --- carry the bulk of the predictive signal, and the tabular models consume them directly while the neural models must rediscover them from raw sequences.

Three threats to validity should temper these conclusions. First, the study uses three buildings at a single anonymized site; while the results align with broad literature, conclusions about which architecture wins at substantially larger scales (e.g.\ city-wide or grid-operator data) cannot be drawn from this work. Second, the held-out test period spans September through December 2017, while the training data is dominated by 2016 and early 2017; the consistent validation-to-test gap observed for every model suggests that seasonal distribution shift contributes non-trivially to the reported test errors, and multi-year data would close that gap. Third, the Transformer encoder is a deliberately compact realization of the attention idea; the full Temporal Fusion Transformer with variable-selection networks and quantile-output heads was not implemented, and may close the gap to the tree models on substantially larger datasets where its additional capacity is justified.

Three concrete future directions emerge naturally. Probabilistic forecasting via quantile regression or conformal prediction \cite{ref_lim_tft} would convert the current point forecasts into interval forecasts more directly useful for unit-commitment and demand-response decisions. Multi-building cross-training (or a federated variant for utilities unwilling to share raw data) would expose the deep architectures to the order-of-magnitude increase in training data they would need to outperform the trees. Finally, a multi-horizon evaluation that compares one-step-ahead accuracy against day-ahead and week-ahead accuracy would test whether the architectural ordering observed here is stable under longer forecasting horizons, where attention-based and decomposition-based methods have stronger theoretical grounding \cite{ref_lim_tft, ref_taylor_short_term}.

The contribution of this work, then, is not to claim that deep learning has been definitively dethroned on electricity-load forecasting --- a stronger claim would require substantially more data and many more buildings --- but rather to provide an honest, reproducible, end-to-end demonstration that, at the scale of a single utility's worth of building meters, the engineered-feature plus gradient-boosting recipe is the right starting point, and that any deep-learning extension to such a system should be evaluated against this recipe rather than against the assumption that deeper models are inherently better.

\section{Broader Reflection}

Beyond the empirical findings, the broader value of this project lies in what the construction process itself taught us about how forecasting systems should be evaluated. The most uncomfortable lesson was that the difference between a credible benchmark and a misleading one often comes down to engineering details that academic papers do not always foreground: whether the train/validation/test split respects time ordering, whether the inference path used for live serving produces the same numbers as the training-time evaluation, whether the metrics module is shared across every model or re-implemented per architecture, and whether the random-seed regime is consistent. Each of these details has the capacity to silently inflate or deflate a leaderboard by an order of magnitude, and none of them is intrinsically a deep-learning concern --- they apply equally to a classical ARIMA pipeline. The discipline of writing a consistency-check harness and treating any disagreement between training and serving as a bug to be fixed (rather than tolerated) was the single most important methodological habit acquired during this work.

We hope that future students using BDG2 will find the released codebase a useful starting point. The data preparation, feature engineering, model definitions, metric implementations, and serving layer are deliberately structured so that adding a tenth model --- whether a different gradient-boosting library, a state-space neural network, or a graph-neural model that exploits cross-building relationships --- requires touching only the model-specific training script and the registry's dispatch table, with the rest of the pipeline left untouched. Reproducibility is not a property of any single component; it is a property of the system, and the system is what we set out to deliver.

\subsection*{E.1 Cyclical feature construction}

The cyclical hour and day-of-year encodings described in Section~3.2 are implemented in \texttt{ml/scripts/build\_features.py} as follows.

{\small\begin{verbatim}
import numpy as np
import pandas as pd

def add_cyclical_features(df: pd.DataFrame) -> pd.DataFrame:
    """Add sin/cos pairs for hour-of-day and day-of-year."""
    h = df["timestamp"].dt.hour
    d = df["timestamp"].dt.dayofyear
    df["hour_sin"] = np.sin(2 * np.pi * h / 24.0)
    df["hour_cos"] = np.cos(2 * np.pi * h / 24.0)
    df["doy_sin"]  = np.sin(2 * np.pi * d / 365.25)
    df["doy_cos"]  = np.cos(2 * np.pi * d / 365.25)
    return df
\end{verbatim}}

\subsection*{E.2 Model registry dispatch}

The serving layer dispatches an incoming \texttt{(model\_name, building\_id, target\_ts)} tuple to the appropriate trained checkpoint. The dispatch is intentionally a flat \texttt{if}-chain rather than a class hierarchy --- each branch is short, the failure mode is clear, and adding a new model takes one diff.

{\small\begin{verbatim}
def predict(self, model_name: str, building_id: str,
            target_ts: pd.Timestamp,
            weather_override: dict | None = None) -> float:
    if model_name in {"random_forest", "xgboost"}:
        return self._predict_tree(model_name, building_id,
                                  target_ts, weather_override)
    if model_name == "arima":
        return self._predict_arima(building_id, target_ts)
    if model_name in {"naive_last", "naive_seasonal_24h"}:
        return self._predict_naive(model_name, building_id, target_ts)
    if model_name in {"lstm", "bilstm", "cnn_lstm", "transformer"}:
        return self._predict_neural(model_name, building_id,
                                    target_ts, weather_override)
    raise ValueError(f"unknown model: {model_name}")
\end{verbatim}}

\subsection*{E.3 Consistency-check harness}

The harness in \texttt{ml/scripts/verify\_consistency.py} draws a random sample of fifty (model, building, timestamp) triples, asks the live API for a prediction for each, and asserts agreement with the stored offline predictions within a fixed tolerance.

{\small\begin{verbatim}
import random, requests, pandas as pd

TOL_KWH = 0.5

def verify(api_base: str = "http://localhost:8000") -> None:
    leaderboard = pd.read_csv("ml/models/results/leaderboard.csv")
    failures = []
    for model in leaderboard["model"]:
        preds = pd.read_parquet(
            f"ml/models/results/predictions/{model}.parquet"
        )
        sample = preds.sample(50, random_state=0)
        for _, row in sample.iterrows():
            r = requests.post(f"{api_base}/forecast", json={
                "model": model,
                "building_id": row["building"],
                "target_datetime": row["timestamp"].isoformat(),
            }).json()
            if abs(r["prediction_kwh"] - row["y_pred"]) > TOL_KWH:
                failures.append((model, row["building"],
                                 row["timestamp"]))
    assert not failures, f"{len(failures)} predictions disagreed"
    print("all consistent")
\end{verbatim}}
